Deploying to Amazon Web Services

A step-by-step guide to moving the entire Chenesa stack from Vercel + Railway + Supabase onto AWS — with architecture diagrams, CLI commands, and cost estimates.

AWS Amplify → Frontend AWS App Runner → Backend AWS ECR → Docker Images Supabase → Keep as-is EventBridge → Scheduler
Section 01

Migration Strategy

There are three levels of AWS migration. Start with Level 1 — it takes ~2 hours and changes nothing about how the app works.

Level 1 — Recommended

Lift & Shift

Move frontend to Amplify, backend to App Runner. Keep Supabase for DB + Auth. Zero code changes needed.

⏱ ~2 hours · $15–35/mo
Level 2 — Partial

Managed Database

Add AWS RDS (PostgreSQL) to replace Supabase DB. Keep Supabase Auth. Requires running SQL migrations.

⏱ ~1 day · $40–80/mo
Level 3 — Full AWS

Full Native AWS

RDS + Cognito Auth + ECS Fargate + CloudFront + WAF + ALB. Enterprise-grade, requires AWS expertise.

⏱ 1–2 weeks · $80–200/mo
✅ Recommendation for now

Start with Level 1. Your backend already has a Dockerfile — that's the hard part done. AWS App Runner reads it directly. Supabase keeps working unchanged. You can always migrate the DB later when you need more control.

Section 05

Step-by-Step Deployment

Follow these steps in order. Total time: ~2 hours for Level 1.

1

Create ECR Repository (Docker image storage)

5 mins

ECR (Elastic Container Registry) stores your backend Docker image. App Runner pulls from it.

# Create the ECR repository
aws ecr create-repository \
  --repository-name chenesa-backend \
  --region us-east-1

# Note the repositoryUri from the output — looks like:
# 123456789.dkr.ecr.us-east-1.amazonaws.com/chenesa-backend
  • Log into the AWS Console → ECR → confirm the repo appears
  • Copy the full URI — you'll need it in step 2
2

Build & Push Backend Docker Image to ECR

15 mins

Your backend already has a Dockerfile in backend/. Build it and push to ECR.

# From the Chenesa project root — replace YOUR_ACCOUNT_ID
ACCOUNT_ID=123456789012
REGION=us-east-1
REPO=${ACCOUNT_ID}.dkr.ecr.${REGION}.amazonaws.com/chenesa-backend

# 1. Authenticate Docker to ECR
aws ecr get-login-password --region ${REGION} \
  | docker login --username AWS --password-stdin ${REPO}

# 2. Build the image (from backend/ folder)
docker build -t chenesa-backend ./backend

# 3. Tag it for ECR
docker tag chenesa-backend:latest ${REPO}:latest

# 4. Push to ECR
docker push ${REPO}:latest
💡 Automate this with GitHub Actions

After manual setup works, add a .github/workflows/deploy-backend.yml that builds + pushes to ECR on every push to main. App Runner can be configured to auto-redeploy when the ECR image changes.

3

Store Secrets in AWS Secrets Manager

10 mins

Move all environment variables out of Railway into AWS Secrets Manager. App Runner will inject them at runtime.

# Store each secret (replace values with your real ones)
aws secretsmanager create-secret \
  --name chenesa/supabase-url \
  --secret-string "https://xxxx.supabase.co"

aws secretsmanager create-secret \
  --name chenesa/supabase-service-key \
  --secret-string "eyJh..."

aws secretsmanager create-secret \
  --name chenesa/anthropic-api-key \
  --secret-string "sk-ant-..."

aws secretsmanager create-secret \
  --name chenesa/google-client-id \
  --secret-string "123456.apps.googleusercontent.com"

aws secretsmanager create-secret \
  --name chenesa/google-client-secret \
  --secret-string "GOCSPX-..."

aws secretsmanager create-secret \
  --name chenesa/frontend-url \
  --secret-string "https://www.yourapp.com"

aws secretsmanager create-secret \
  --name chenesa/cron-secret \
  --secret-string "your-random-secret"
  • Cost: $0.40 per secret per month = ~$3/mo total — very cheap
  • All secrets are encrypted with KMS at rest automatically
4

Deploy Backend on AWS App Runner

20 mins

App Runner is the AWS equivalent of Railway — it runs your Docker container with auto-scaling and zero cluster management.

# First create an IAM role for App Runner to pull from ECR
# Go to: AWS Console → IAM → Roles → Create Role
# Trusted entity: App Runner → Add policy: AmazonEC2ContainerRegistryReadOnly
# Name it: AppRunnerECRAccessRole

# Then in AWS Console → App Runner → Create Service:
#   Source: Container registry → Amazon ECR
#   Image URI: 123456789.dkr.ecr.us-east-1.amazonaws.com/chenesa-backend:latest
#   Deployment trigger: Automatic (redeploys when image updates)
#   Port: 8000
#   CPU: 0.25 vCPU  |  Memory: 0.5 GB  (cheapest, scales up if needed)
#   Health check: GET /health  (add this endpoint to FastAPI if not exists)

Then add environment variables in the App Runner console (Environment variables section):

# In App Runner → Configuration → Environment variables
SUPABASE_URL        = (from Secrets Manager)
SUPABASE_SERVICE_KEY = (from Secrets Manager)
ANTHROPIC_API_KEY   = (from Secrets Manager)
GOOGLE_CLIENT_ID    = (from Secrets Manager)
GOOGLE_CLIENT_SECRET = (from Secrets Manager)
FRONTEND_URL        = https://www.yourapp.com
CRON_SECRET         = (from Secrets Manager)
PORT                = 8000
  • App Runner gives you a URL like abc123.us-east-1.awsapprunner.com
  • Update Google OAuth console: add this URL to Authorised redirect URIs
  • Test: visit https://abc123.us-east-1.awsapprunner.com/docs — you should see FastAPI Swagger
💡 Update Google OAuth Redirect URI

Go to Google Cloud Console → APIs & Services → Credentials → your OAuth client and add your new App Runner URL to the Authorised redirect URIs list: https://abc123.us-east-1.awsapprunner.com/auth/gmail/callback

5

Deploy Frontend on AWS Amplify

20 mins

Amplify reads your GitHub repo, detects Next.js automatically, and builds + deploys on every push — exactly like Vercel.

# Option A: Via AWS Console (recommended first time)
# AWS Console → Amplify → Create new app → GitHub
# Select repo: Luke-Manyamazi/Chenesa
# Branch: main
# Root directory: frontend
# Framework: Next.js (auto-detected)

# Option B: Via Amplify CLI
cd frontend
amplify init
amplify add hosting
# Select: Hosting with Amplify Console
# Select: Continuous deployment (Git-based)
amplify publish

Set environment variables in Amplify Console → App settings → Environment variables:

# Amplify → App settings → Environment variables
NEXT_PUBLIC_SUPABASE_URL      = https://xxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY = eyJh...
NEXT_PUBLIC_API_URL           = https://abc123.us-east-1.awsapprunner.com
  • Amplify gives you a URL like main.abc123.amplifyapp.com
  • Add a custom domain in Amplify → Domain management → Add domain
  • SSL certificate is issued automatically via ACM (free)
✅ Next.js SSR fully supported

Amplify handles Next.js Server Components, API routes, and middleware correctly. It uses Lambda@Edge under the hood for SSR — no configuration needed.

6

Configure Custom Domain with Route 53

15 mins

Point your domain (e.g., chenesa.com) at AWS. Route 53 handles DNS; ACM issues the SSL cert automatically.

# 1. Register or transfer domain in Route 53 (or use existing registrar)
# AWS Console → Route 53 → Hosted zones → Create hosted zone
# Domain: chenesa.com

# 2. Add a CNAME for your frontend
# Record: www → main.abc123.amplifyapp.com
# (Amplify will also verify and issue the cert automatically)

# 3. Add a CNAME for your backend API
# Record: api → abc123.us-east-1.awsapprunner.com
# (Use custom domain in App Runner too for clean URLs)

# 4. Update App Runner custom domain
aws apprunner associate-custom-domain \
  --service-arn arn:aws:apprunner:... \
  --domain-name api.chenesa.com
  • Route 53 hosted zone: $0.50/mo per zone
  • DNS queries: $0.40 per million queries
  • ACM certificates: free when used with AWS services
7

Update Environment Variables

5 mins

Now that both services have real URLs, wire them together.

# In Amplify → Environment variables, update:
NEXT_PUBLIC_API_URL = https://api.chenesa.com

# In App Runner → Environment variables, update:
FRONTEND_URL = https://www.chenesa.com

# In Google Cloud Console → OAuth credentials, update redirect URI:
https://api.chenesa.com/auth/gmail/callback

# In GitHub Actions secret BACKEND_URL, update:
https://api.chenesa.com
  • Redeploy both services after updating env vars
  • Test Gmail OAuth end-to-end with the new URLs
  • Test a cleaning run to confirm Anthropic API calls still work

(Optional) Replace GitHub Actions Scheduler with EventBridge + Lambda

Optional · 30 mins

GitHub Actions already works and is free. Only do this if you want everything in AWS.

# Create a Lambda function that calls the backend scheduler
# Lambda → Create function → Author from scratch
# Runtime: Python 3.12

# Lambda code (paste into inline editor):
import urllib.request, os, json

def handler(event, context):
    url = os.environ['BACKEND_URL'] + '/scheduler/trigger'
    req = urllib.request.Request(url, method='POST')
    req.add_header('x-scheduler-secret', os.environ['CRON_SECRET'])
    urllib.request.urlopen(req, timeout=30)

# EventBridge → Rules → Create rule
# Schedule: cron(0 */6 * * ? *)   ← every 6 hours
# Target: your Lambda function

# Set Lambda env vars:
# BACKEND_URL = https://api.chenesa.com
# CRON_SECRET = your-cron-secret
  • EventBridge rule: free for scheduled events
  • Lambda invocations: first 1M per month free forever
  • Total cost for scheduler: $0/mo
Section 06

Auto-Deploy CI/CD Pipeline

Set this up so every git push origin main automatically deploys both services — same as the current Vercel + Railway workflow.

▲ Frontend (Amplify)

Amplify connects directly to GitHub — no workflow file needed. Just enable in the console:

1
Amplify → App settings → General
Connect GitHub → select Chenesa repo → branch: main → root: frontend/
2
Set build settings
Amplify auto-detects Next.js and generates correct build config
3
Done — auto-deploys on every push
~3 min build time. Preview deploys for PRs are free too.

🚂 Backend (ECR + App Runner)

Add this GitHub Actions workflow file to auto-build and push the Docker image:

# .github/workflows/deploy-backend.yml
name: Deploy Backend
on:
  push:
    branches: [main]
    paths: ['backend/**']

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Configure AWS
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_KEY }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET }}
          aws-region: us-east-1
      - name: Build & Push to ECR
        run: |
          aws ecr get-login-password | \
            docker login --username AWS \
            --password-stdin $ECR_URI
          docker build -t $ECR_URI:latest ./backend
          docker push $ECR_URI:latest
        env:
          ECR_URI: ${{ secrets.ECR_URI }}
💡 App Runner auto-redeployment

In App Runner settings, enable "Automatic deployment" — it watches ECR and redeploys within 2 mins when a new image is pushed.

Section 08

Level 2 — Adding AWS RDS (Optional Later)

When you need more DB control, compliance requirements, or >500 MB data, migrate from Supabase to RDS PostgreSQL.

Migrate DB to RDS

# 1. Launch RDS PostgreSQL in AWS Console
# Engine: PostgreSQL 15
# Instance: db.t3.micro (~$15/mo)
# Storage: 20 GB gp2
# Multi-AZ: No (enable for prod HA)
# VPC: same as App Runner

# 2. Export Supabase data
pg_dump "postgres://postgres:pass@\
  xxxx.supabase.co:5432/postgres" \
  > chenesa_backup.sql

# 3. Import to RDS
psql "postgres://admin:pass@\
  rds-endpoint.us-east-1.rds.amazonaws.com\
  /chenesa" < chenesa_backup.sql

# 4. Update App Runner env var
DATABASE_URL = postgres://admin:pass@rds-endpoint...

Keep Supabase Auth

Even with RDS for storage, keep Supabase Auth for JWTs. It saves you from building user management, email verification, password reset, MFA, and OAuth from scratch.

✓Supabase Auth works with any PostgreSQL DB
✓Keep using JWT validation in FastAPI middleware
✓User profile data can live in RDS, auth in Supabase
✗Migrating to Cognito = rewriting all auth code (not worth it)

Security Checklist

Things to lock down before going live on AWS.

🔐 Must Do

□Enable MFA on AWS root account
□Never commit AWS credentials to GitHub
□Never use root account for deployments
□Restrict IAM user to only needed services
□Add AWS Billing Alert at $50 threshold
□All secrets in Secrets Manager — no hardcoded values
□Update Google OAuth redirect URIs to new App Runner URL
□Remove old Railway URL from Google OAuth

✅ Nice to Have

□Enable AWS GuardDuty (threat detection, ~$5/mo)
□Enable AWS CloudTrail (audit all API calls)
□Add WAF in front of App Runner (rate limiting)
□Use VPC for App Runner + private subnets
□Enable RDS encryption at rest (if using RDS)
□Rotate API keys every 90 days
□Set App Runner auto-scaling max to prevent cost spikes
