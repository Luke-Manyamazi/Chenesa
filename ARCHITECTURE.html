Email Categories Claude Assigns

🚫 SPAM

Phishing, fake prizes, unsolicited commercial email. Highest priority delete.

📢 MARKETING

Sales newsletters, promotional offers, flash sales, brand announcements.

🌐 SOCIAL

Facebook, LinkedIn, Twitter notifications. Low-signal social alerts.

📅 OLD_READ

Read and older than threshold. Pre-filtered — never sent to Claude AI.

✅ KEEP

Invoices, receipts, personal mail, bank statements. Always preserved.

❓ UNKNOWN

Fallback when API fails or Claude is uncertain. Never deleted — safe default.

Section 04

Gmail OAuth 2.0 + PKCE Flow

How a user's Gmail account gets connected securely. Railway handles the OAuth dance server-side.

1 → User clicks "Connect Gmail" Vercel calls GET /auth/gmail/url on Railway
2 → Railway generates OAuth URL + PKCE google_auth_oauthlib auto-creates code_verifier → stored in _verifier_store[state]
3 → Browser redirected to Google consent screen User approves Gmail read + delete scopes
4 → Google calls Railway callback with ?code=…&state=… chenesa-production.up.railway.app/auth/gmail/callback
5 → Railway exchanges code → tokens Pops code_verifier from store, calls flow.fetch_token(code, code_verifier)
6 → Railway saves encrypted tokens to Supabase email_accounts access_token + refresh_token stored encrypted
7 → Railway redirects browser to Vercel 302 → chenesa.vercel.app/accounts?connected=user@gmail.com

Section 06

Claude AI — Classifier Internals

How the email classification engine is built for speed and cost-efficiency.

⚡ Prompt Caching

The 500-token system prompt is marked cache_control: ephemeral. On a 500-email run (25 API calls):

Call #1 — cache WRITE 1.25× cost
Calls #2–25 — cache HIT 0.1× cost each
Effective average ≈ 0.15× cost

Cache key is stable because the system prompt is a module-level constant — no timestamps, no per-user data injected.

📦 Batch Processing

Up to 20 emails are sent in a single API call. Claude returns a JSON array — one result per email:

[
  {
    "email_num": 1,
    "category": "MARKETING",
    "confidence": "high",
    "reasoning": "Flash sale newsletter"
  },
  { "email_num": 2, "category": "KEEP", ... }
]

MAX_TOKENS = 4096 — each result needs ~100 tokens; batch of 20 needs ~2,000 tokens minimum.

🛡 Safety Fallbacks

Multiple layers prevent accidental deletion:

✓API failure → all emails marked UNKNOWN (never deleted)
✓JSON parse error → fallback to UNKNOWN
✓Missing batch items → defensive per-email fallback
✓Keep rules override ALL AI decisions
✓Dry-run mode: simulate without deleting
✓Retry on 429 / 5xx with exponential backoff (tenacity)

🔁 Rate Limiting

A shared token-bucket RateLimiter controls API request cadence across concurrent cleaning runs:

Anthropic bucket
Acquired before every messages.create() call
IMAP bucket
Acquired before every mailbox operation

Model: claude-haiku-4-5-20251001 — fast, cheap, accurate for classification.

Section 08

Railway API Endpoints

All authenticated with Supabase JWT in the Authorization: Bearer <token> header, except the OAuth callback.

POST /clean/{account_id} Trigger a cleaning run for one account
POST /clean/all-accounts GitHub Actions: clean every enabled account
GET /auth/gmail/url Generate OAuth consent URL + store PKCE verifier
GET /auth/gmail/callback Google redirects here · exchange code → tokens · 302 to Vercel
GET /accounts List user's connected email accounts
POST /accounts/imap Connect Yahoo / Outlook / iCloud via IMAP + app password
DEL /accounts/{id} Disconnect and delete an account
GET /history/runs Cleaning run history with status + counts
GET /history/runs/{id} Full email-level detail for one run
GET /settings/keep-rules List user's keep rules
POST /settings/keep-rules Add a new keep rule (keyword + match field)
DEL /settings/keep-rules/{id} Remove a keep rule
GET /settings/profile Get user preferences (old_read_days)
PATCH /settings/profile Update cleaning aggressiveness threshold
